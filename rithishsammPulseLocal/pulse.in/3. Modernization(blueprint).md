Managing deployments across multiple clusters using Git as the source of truth. For - Hybrid setup → multiple clusters and Manual deployment = inconsistent. 

Architectural Deployment Diagram:

Replacing:
![[Not recommended CICD Workflow.png]]

with:
![[Modern Dual Pipeline CICD + Feedback Loop Workflow.png]]


> **This might seem very specific and maybe unconventional than traditional CI/CD Pipelines. This is planned because,** 
> 	1. ==We get constant, frequent feedback loop support tickets and feature request where we fix the same right away with minimal to no checking hops. ==
> 	2. ==Since it has been consistent and work well for us for our specific usecase, no matter what QA/QC is unavoidable and highly necessary for any means. For such case, came up with ==
> 	3. ==Feedback Loop Pipeline - where it goes through all the checks and provides with all the security reports, CVE lists, Code Coverage and more in detail for each deployment in a unified visibility.==

# Flow Explanation (Simple)

1. Client → AWS (Signaling/API)
2. AWS → decides session/media routing
3. Media traffic → On-Prem (RTP cluster)
4. Control remains in AWS
5. GitOps controls both environments centrally.
